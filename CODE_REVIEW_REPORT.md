## 🔐 1. Аутентификация и управление сессиями (`app/api/auth.py`, `app/services/auth_service.py`)

### 👍 Сильные стороны
* **Гибридный подход:** Отличная поддержка как локальных пользователей (пароли + bcrypt), так и OAuth/OTP пользователей через Supabase Auth.
* **Локальный режим для тестов (`LOCAL_MODE`):** Умное переключение на моки при отсутствии конфигурации SMTP или во время тестов в pytest. Это ускоряет разработку и предотвращает лишние траты ресурсов.

### 🚨 Критическая уязвимость: Декодирование JWT без проверки подписи (High Severity)
В файле `app/api/auth.py` в функции `get_current_user` для пользователей Supabase JWT (Path 2) подпись токена **не проверяется**:

```python
# Линия 57-58 в app/api/auth.py
payload_b64 = parts[1] + '=' * (-len(parts[1]) % 4)
payload = json.loads(base64.urlsafe_b64decode(payload_b64))
email = payload.get('email')
```
* **В чём опасность:** Любой злоумышленник может сгенерировать фейковый JWT-токен на стороне клиента, вписать туда email любого зарегистрированного пользователя (например, администратора) и отправить его в заголовке `Authorization`. Бэкенд успешно декодирует его и авторизует злоумышленника в системе.
* **Как исправить:** 
  JWT обязательно нужно валидировать. Поскольку это JWT от Supabase, его нужно проверять с помощью секрета JWT вашего проекта Supabase (доступен в панели Supabase как `JWT Secret`), либо использовать встроенный метод клиента Supabase, который делает запрос к API Supabase для валидации токена.

**Рекомендуемый безопасный вариант проверки:**
```python
# Безопасная проверка через клиент Supabase:
try:
    # Запрос к Supabase для верификации токена и получения данных пользователя
    user_resp = google_auth_service.client.auth.get_user(token)
    if not user_resp or not user_resp.user:
        raise HTTPException(status_code=401, detail="Неверный или истекший токен")
    
    email = user_resp.user.email
    # Дальнейшая работа с локальным профилем...
except Exception as e:
    raise HTTPException(status_code=401, detail=f"Ошибка авторизации токена: {str(e)}")
```

## 📄 2. Обработка документов и AI-интеграция (`app/services/ai_service.py`, `app/services/file_service.py`)

### 👍 Сильные стороны
* **Казахстанский контекст:** Использование юридических стандартов РК (ГК РК, ТК РК, тенге) в промпте `SYSTEM_PROMPT`. Это делает результаты невероятно точными для локального рынка.
* **Стабильный JSON:** Передача параметра `response_format={"type": "json_object"}` в OpenAI-клиент гарантирует стабильную структуру ответа от AI без мусорного текста.
* **Идеальный UX-фолбэк:** Наличие демо-анализа и мягкой обработки ошибок. Если у пользователя закончился баланс на HuggingFace или нет сети, приложение не падает, а выдает красивый пример анализа с пояснением, как настроить токен.

### ⚠️ Проблема 1: Жесткое урезание текста договора до 15 000 символов (Medium Severity)
В файле `ai_service.py` текст обрезается:
```python
text_to_analyze = contract_text[:15000]
```
* **В чём опасность:** 15,000 символов — это всего около 2000-2500 слов (3-4 страницы договора). Большинство реальных договоров занимают от 10 страниц. Вся важная информация об ответственности сторон, расторжении и штрафах обычно находится в самом конце договора, который при такой реализации **полностью отрезается**.
* **Как исправить:** Модель Kimi K2.5 поддерживает контекст в 128k+ токенов. Вы можете смело увеличить лимит до **100 000 символов** без потери качества, либо использовать интеллектуальную фильтрацию (вырезать пустые строки, реквизиты в конце и т.д.).

### 💡 Проблема 2: Отсутствие OCR-фолбэка для отсканированных PDF (Low-Medium Severity)
Метод `FileService.extract_text_from_pdf` извлекает только встроенный текст с помощью `PyMuPDF`:
```python
page_text = page.get_text("text")
```
* **В чём проблема:** Если пользователь загрузит отсканированный PDF (файл, созданный из картинок-страниц), `page.get_text()` вернет пустую строку, и сервис выдаст ошибку: *"Текст не найден в документе"*.
* **Как исправить:** Добавить автоматическое переключение на OCR (Tesseract), если при обычном извлечении текста из PDF вернулось менее 100 символов:

```python

full_text, page_count = cls.extract_text_from_pdf_raw(file_bytes)
if len(full_text.strip()) < 100:
```

## 🗄️ 3. База данных, миграции и лимиты (`supabase_migration.sql`, `app/services/usage_limiter.py`)

### 👍 Сильные стороны
* **Защита от состояния гонки (Race Condition):** Настоящая жемчужина вашей базы данных — это атомарная SQL-функция `increment_analyses_used`. Блокировка строки `FOR UPDATE` исключает баг, когда быстрый двойной клик пользователя позволял обойти лимиты тарифа.
* **Безопасная работа с соединениями:** Все операции в `UsageLimiter` защищены конструкциями `try...finally: db.close()`, что полностью исключает утечки пула подключений к PostgreSQL.
* **Интеграция с Supabase Auth:** Использование триггера `trg_on_auth_user_created` для авто-создания локального профиля пользователя при регистрации на стороне Supabase — это золотой стандарт разработки с Supabase.

### 📝 Замечание: Извлечение Email без проверки подписи для лимитов
В `usage_limiter.py` в функции `_decode_jwt_email` используется декодирование JWT без проверки подписи. 
* **Это допустимо?** **Да, здесь это безопасно**, так как функция используется исключительно для фолбэк-трекинга лимитов (если у пользователя истек токен) и не дает доступа к защищенным данным или действиям от имени пользователя. Код содержит четкий комментирующий ворнинг, что заслуживает отдельного плюса.
